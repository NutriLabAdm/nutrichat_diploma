## 0. Едичные справочники

Перед написанием текста всегда сверяться со справочниками:

| Справочник | Файл | Назначение |
|------------|------|------------|
| Термины | `00_Справочники/Термины.md` | Основные определения платформы |
| Сервисы | `00_Справочники/Сервисы.md` | Экосистема, URL, статусы |
| Глоссарий | `00_Справочники/Глоссарий.md` | Рус/Eng соответствие |

## 1. Общие требования к структуре глав

### 1.1. Обязательная структура каждого раздела (подраздела)

Каждый подраздел главы должен содержать:

```
### [Номер]. [Название подраздела]

**Предпосылки:** [Текстовый абзац с объяснением, почему возник этот подраздел, из каких предпосылок он появился]

**Цели подраздела:** [Текстовый абзац с целями, для чего нужен этот подраздел, какие задачи решает]

[Основное содержание подраздела]

*[При наличии] Ссылка на приложение: Приложение X.X.*
```

### 1.2. Нумерация глав и подразделов

- Главы: арабские numerals (1, 2, 3, ...)
- Подразделы: десятичная нумерация (1.1, 1.2, ..., 2.1, 2.2, ...)
- Приложения: Приложение X.X

### 1.3. Структура Введения

Введение должно содержать:
- Актуальность темы исследования
- Цель исследования
- Задачи исследования (пронумерованные)
- Объект исследования
- Предмет исследования
- Методы исследования
- Научная новизна
- Практическая значимость
- Структура работы (таблица глав)

## 2. Требования к приложениям

### 2.1. Именование приложений

Формат: `Приложение [X.X] [Название].md`

Примеры:
- `Приложение 2.4 UseCases.md`
- `Приложение 2.5 Трассируемость.md`
- `Приложение 2.7 C4 Model.md`
- `Приложение 2.8 Метрики.md`

### 2.2. Обязательные элементы приложения

- Заголовок с номером и названием
- Раздел "Введение" с описанием назначения
- Основное содержание
- Дата создания

### 2.3. Когда создавать приложение

Приложение создаётся, когда:
- Материал слишком объёмный для основного текста
- Нужны диаграммы или схемы
- Требуется детальная таблица или матрица
- Информация может быть полезна отдельно

### 2.4. Ссылки на приложения

В основном тексте:
```
*[Детальная информация — в Приложении X.X.]*
```

## 3. Требования к диаграммам и изображениям

### 3.1. Расположение диаграмм

- Mermaid диаграммы — непосредственно в тексте .md файла
- PlantUML/диаграммы изображений — в папке `Приложения/images/[номер главы]/`
- Формат: `.svg` (предпочтительно) или `.png`

### 3.2. Структура папок для изображений

```
Приложения/
└── images/
    ├── 1/            # Глава 1
    ├── 2/            # Глава 2
    ├── 3/            # Глава 3
    ├── 4/            # Глава 4
    ├── 5/            # Глава 5
    └── ...
```

**Правило:** Все изображения хранятся в `Приложения/images/[номер главы]/`

### 3.3. Формат именования файлов изображений

**Структура имени файла:**
```
{номер главы}.{номер секции раздела в главе}_{название изображения}.[mmd|png|svg|puml]
```

**Где:**
- `{номер главы}` — номер главы (1, 2, 3, ...)
- `{номер секции раздела в главе}` — номер подраздела внутри главы (1, 2, 3, ...)
- `{название изображения}` — название изображения, диаграммы (snake_case)
- Расширения: `.mmd` (Mermaid), `.png`, `.svg`, `.puml` (PlantUML)

**Примеры:**
- `1.1_dev_process.svg` — Глава 1, раздел 1, диаграмма процесса разработки
- `2.7_c4_containers.svg` — Глава 2, раздел 7, C4 контейнеры
- `3.2_data_flow.svg` — Глава 3, раздел 2, поток ��анных
- `4.2_ai_chat.png` — Глава 4, раздел 2, AI чат UI
- `5.1_ab_test_results.png` — Глава 5, раздел 1, результаты A/B тестов

**Примеры с исходниками:**
- `3.2_er_diagram.puml` → `3.2_er_diagram.svg`
- `2.7_c4_containers.mmd` → `2.7_c4_containers.svg`

### 3.4. Именование диаграмм в тексте

Формат: `[номер]_[название].svg`

Примеры:
- `c4_context.svg`
- `01_stakeholders.png`
- `02_architecture.svg`

### 3.4. Генерация диаграмм

Инструменты для генерации диаграмм:

**Mermaid (рекомендуется):**
```bash
mmdc -i input.mmd -o output.svg -w [width]
```

**PlantUML (локальный):**
```bash
java -jar plantuml.jar -tsvg input.puml
java -jar plantuml.jar -tpng input.puml
```

**Через скрипты в `99_tools/`:**
```bash
python 99_tools/mermaid_converter.py input.mmd output.svg
python 99_tools/plantuml_converter.py input.puml output.svg
```

### 3.5. Запрет на эмоджи

**Правило:** Запрещено использовать эмоджи в тексте документов, названиях файлов и ссылках.

**Почему:** Эмоджи могут некорректно отображаться в разных ОС и редакторах, затрудняют поиск и автоматизацию.

**Примеры запрещённых:** [clipboard] [gear] [microscope] [test] [game] [chart] [heart] [DNA]

**Альтернативы:** Использовать текстовые обозначения или Unicode-символы.

### 3.7. Запрет на псевдографику

**Правило:** Запрещено использовать псевдографику (ASCII-art, рамки из симолов ┌─┐ и т.д.) для отображения интерфейсов, схем и диаграмм.

**Почему:** Псевдографика плохо конвертируется в PDF, может отображаться некорректно в разных редакторах, затрудняет автоматическую генерацию документации.

**Альтернатива:** Использовать SVG-диаграммы в папке `Приложения/images/[номер главы]/`

**Примеры преобразования псевдографики в SVG:**
- Интерфейсы → SVG-схемы
- Схемы компонентов → Mermaid/PlantUML → SVG
- Таблицы → Диаграммы в draw.io/PlantUML

**Конвертация SVG в PNG:**
```bash
python 99_tools/svg2png_converter.py input.svg output.png
python 99_tools/svg2png_converter.py input.svg --scale=2
```

*Детальная информация — в `99_tools/README.md`*

### 3.8. Обязательные элементы диаграммы

- Заголовок с названием (caption)
- Описание в тексте после диаграммы
- Использование цветов по назначению:
  - `#e3f2fd` — Пользователи
  - `#bbdefb` — Внутренние системы
  - `#90caf9` — Внешние системы
  - `#e8f5e9` — Auth/Data сервисы

## 4. Обеспечение консистентности

### 4.1. Терминология

**Правило:** Использовать русский термин, если есть эквивалент. При отсутствии русского термина — английский с переводом в скобках.

**Примеры:**

| Английский | Русский | Notes |
|------------|---------|-------|
| latency | задержка (отклик) | "время отклика" чаще в контексте API |
| scalability | масштабируемость | |
| downtime | простой | |
| throughput | пропускная способность | |
| rollback | откат | |
| monitoring | мониторинг | |
| orchestration | оркестрация | |
| CI/CD | Непрерывная интеграция/доставка (CI/CD) | Английский чаще |
| fallback | резервный вариант | |
| endpoint | конечная точка (endpoint) | |
| container | контейнер | Docker |
| thread | поток | |
| daemon | фоновая служба (daemon) | |
| pipeline | конвейер (pipeline) | |
| token | токен | JWT tokens |
| benchmark | контрольные замеры (бенчмарк) | |
| snapshot | снимок (state) | |

### 4.2. Борьба с дублированием

**Правило:** Избегать дублирования таблиц, диаграмм и текста между главами.

Принципы:
1. **Единый источник** — диаграммы уровня C4 (Level 1, Level 2) создаются в одном месте (Приложения/images/)
2. **Кросс-ссылки** — ссылаться на приложения из других глав, не дублировать
3. **Принцип DRY** — Don't Repeat Yourself

**Примеры:**

| Элемент | Где определено | Где используется |
|---------|---------------|------------------|
| C4 Level 1 System Context | Глава 2, Приложение 2.7 | Ссылки во всех главах |
| C4 Level 2 Containers | Глава 3, Приложение 3.1 | Глава 3 |
| ER-диаграмма | Глава 3, Приложение 3.2 | Глава 3, Приложение 3.2 |
| Архитектура Digital Twin | Глава 3, Приложение 3.3 | Глава 3, Глава 5 |

**Порядок действий:**
1. Создать диаграмму/таблицу в первой главе, где она логически обоснована
2. Добавить в соответствующее Приложение с номером главы
3. В других главах использовать ссылку: *Детальная диаграмма — в Приложении X.X*

### 4.3. Проверка консистентности документов

**Чек-лист перед коммитом:**
- [ ] Нет эмоджи в тексте
- [ ] Нет эмоджи в названиях файлов
- [ ] Нет псевдографики
- [ ] Изображения имеют подпись "Рисунок X.X"
- [ ] Все ссылки на изображения работают
- [ ] Нумерация подразделов последовательна
- [ ] Ссылки на приложения в формате "*Детальная информация — в Приложении X.X*"

### 4.4. Консистентность ссылок на изображения

**Правило:** Все ссылки на изображения должны быть консистентными во всех документах, главах и приложениях.

**Формат ссылки на изображение:**
```
Рисунок X.X — [Название изображения]
```

**Примеры:**
```
Рисунок 2.7 — Диаграмма контейнеров C4
Рисунок 3.2 — ER-диаграмма системы
Рисунок 5.1 — Результаты A/B тестирования
```

**Прямая ссылка на файл:**
```
Рисунок X.X — [Название изображения] (Приложения/images/[номер главы]/[имя файла].svg)
```

**Порядок действий при добавлении изображения:**
1. Разместить файл в `Приложения/images/[номер главы]/`
2. Соблюдать формат именования (раздел 3.3)
3. Добавить подпись в тексте: `Рисунок X.X — [Название]`
4. Проверить, что изображение не дублирует существующее
5. При необходимости сослаться на существующее изображение из другой главы

**Запрещено:**
- Создавать дубликаты изображений в разных папках
- Использовать разные форматы именования для одного типа изображений
- ссылаться на изображения без подписи

**Проверка консистентности:**
- [ ] Все изображения имеют подпись в формате `Рисунок X.X — [Название]`
- [ ] Имена файлов соответствуют формату (раздел 3.3)
- [ ] Нет дубликатов в разных папках
- [ ] Пути к файлам корректны
- [ ] Приложения ссылаются на правильные файлы
- [ ] Все изображения соответствуют новой структуре именования

## 7. Актуализация требований

### 7.1. Когда обновлять документ

Документ должен обновляться при:
- Добавлении новой главы
- Изменении структуры существующих глав
- Добавлении новых сервисов в экосистему
- Изменении статусов сервисов
- Создании новых приложений
- Изменении нумерации разделов
- При работе с AI (все промпты должны учитывать эти правила)

### 7.2. Процедура актуализации

1. Проверить актуальность справочника терминов (раздел 4)
2. Проверить состав экосистемы (раздел 5)
3. Обновить статус при изменениях
4. Добавить новые сервисы/модули
5. Обновить дату и версию документа

### 7.3. Требования к промптам

При работе с AI (генерация текста, диаграмм, приложений) всегда:
- Ссылаться на этот документ
- Проверять консистентность терминов
- Использовать правильный путь к изображениям
- Соблюдать формат именования
